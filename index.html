Case Studies

Case Study 1: Logical Flaw in Account Creation Confuses Email with Username

Executive Summary:

A logical flaw was discovered in the user registration process, where the system incorrectly applied username validation rules to the email address field. This resulted in a denial of service for new users, as many common email addresses were rejected, making it impossible to create an account.

Technical Details:

  • Vulnerability Type: Logical Flaw / Improper Authentication (Generic).
  • Description: The account creation form's validation logic was misconfigured. It was applying a strict, non-standard character and length limit (max 24 characters) to the email field, confusing it with the username field. This broke the registration process for valid email addresses.
  • Proof of Concept (PoC):
    1. Attempt to create a new account with a standard email address containing special characters or one longer than 24 characters.
    2. The form will either clear itself or return a generic error about invalid characters or an incorrect length, preventing account creation.

Case Study 2: Information Disclosure via Broken Access Control on a Technical Drawing

Executive Summary:

A broken access control vulnerability was discovered, allowing unauthorized users to directly access sensitive technical drawings and internal documents by manipulating public-facing URLs. This vulnerability could lead to the unauthorized disclosure of proprietary information and internal data.

Technical Details:

  • Vulnerability Type: Broken Access Control / Insecure Direct Object Reference (IDOR).
  • Description: The application did not properly validate user permissions when accessing files stored in a specific directory. By observing the URL pattern of a single public PDF document, I was able to predict and access other sensitive files, suggesting a broader, systemic folder permission issue.
  • Proof of Concept (PoC):

    I accessed a publicly available PDF document with a specific URL format. By changing the document identifier in the URL, I was able to successfully access and view other restricted technical drawings without any authentication or authorization. This indicates a failure in the application's access control logic to prevent unauthorized direct object access.

Case Study 3: IDOR on User Profile Endpoint leads to 500 Internal Server Error

Executive Summary:

An Insecure Direct Object Reference (IDOR) vulnerability was found on a user profile API endpoint. Instead of returning a proper authorization error, the server crashed with a 500 Internal Server Error, indicating a flaw in the access control logic that could be used to cause a denial of service.

Technical Details:

  • Vulnerability Type: Insecure Direct Object Reference (IDOR).
  • Endpoint: /systemcenterapi/v1/users/
  • Description: When attempting to access another user's profile by changing the ID in the API request, the server responded with a 500 Internal Server Error. This confirms a broken access control issue where the application fails to correctly handle an unauthorized request, leading to a server crash instead of a proper 403 Forbidden response.
  • Proof of Concept (PoC):
    1. Log in as a valid user.
    2. Capture the API request to your own profile, which contains your user ID.
    3. Change the user ID in the request path to a different, valid user ID.
    4. The server will respond with a 500 Internal Server Error, indicating that the request was improperly processed and caused a server-side crash.

Case Study 4: Overly Aggressive Security Measures Lead to Denial of Service

Executive Summary:

A denial of service vulnerability was identified where a simple, harmless search query was being misinterpreted as a malicious attack. The overly aggressive security system immediately blocked the user's IP, preventing legitimate access to the site. This could lead to a poor user experience and a loss of business.

Technical Details:

  • Vulnerability Type: Denial of Service (DoS).
  • Endpoint: Search functionality
  • Description: The Web Application Firewall (WAF) or other security controls on the search page were too sensitive. A basic HTML payload, such as `

    XSS

    `, was treated as a malicious attempt, causing the user's IP to be completely blocked from the site.
  • Proof of Concept (PoC):
    1. Navigate to the search page.
    2. Type a non-malicious HTML payload (e.g., `

      XSS

      `) into the search bar.
    3. Submit the query. The security system will block your IP, and you will receive a "You have been blocked" error message, effectively causing a denial of service.
